# BLC Care Prayer Parser Rules

## 1. 문서 개요

본 문서는 BLC Care의 기도제목 일괄 입력 및 자동 분리 기능의 규칙을 정의한다.

이 기능은 셀리더가 여러 사람의 기도제목을 한 번에 붙여넣으면, 시스템이 사람별로 자동 분리하여 각 성도의 기도제목 입력칸에 반영하는 기능이다.

MVP에서는 외부 AI API를 사용하지 않고, 규칙 기반 파서로 구현한다.

---

# 2. 기능 목적

셀리더는 실제 운영에서 다음과 같은 형태로 기도제목을 받는 경우가 많다.

```txt
지현 : 회사 일이 잘 마무리되도록 기도해주세요.
진수 : 가족 건강을 위해 기도해주세요.
주원 : 시험 준비를 위해 기도해주세요.
```

BLC Care는 이 텍스트를 다음 구조로 변환한다.

| 입력 이름 | 매칭 성도 | 기도제목                   |
| ----- | ----- | ---------------------- |
| 지현    | 장지현   | 회사 일이 잘 마무리되도록 기도해주세요. |
| 진수    | 김진수   | 가족 건강을 위해 기도해주세요.      |
| 주원    | 박주원   | 시험 준비를 위해 기도해주세요.      |

---

# 3. 기본 원칙

## 3.1 저장 전 확인 원칙

자동 분리 결과는 바로 DB에 저장하지 않는다.

반드시 사용자가 분리 결과를 확인한 뒤 저장한다.

처리 흐름:

```txt
일괄 입력
 → 자동 분리
 → 결과 확인
 → 수정 또는 후보 선택
 → 리포트 저장 시 반영
```

---

## 3.2 선택된 셀 기준 매칭

기도제목 이름 매칭은 전체 성도 DB가 아니라 **선택된 셀의 인원 목록**을 기준으로 한다.

예:

* `/reports/new`에서 1셀 선택
* 기도제목 일괄 입력
* 1셀 인원 중에서만 이름 매칭

Admin이 전체 셀 리포트를 작성하는 경우에도, 반드시 특정 셀을 먼저 선택한 뒤 해당 셀 인원을 기준으로 매칭한다.

---

## 3.3 개인정보 보호 원칙

MVP에서는 외부 AI API에 기도제목을 전송하지 않는다.

기도제목은 민감한 개인정보일 수 있으므로 다음 원칙을 따른다.

* 외부 AI API 사용 금지
* 프론트엔드 또는 Apps Script 내부 규칙 기반 처리
* 저장 전 사용자 확인 필수
* 원본 입력 텍스트는 필요한 경우에만 저장

---

# 4. 입력 형식

## 4.1 기본 지원 형식

아래 형식을 우선 지원한다.

```txt
이름 : 기도제목
이름: 기도제목
이름 - 기도제목
이름) 기도제목
이름/ 기도제목
```

예시:

```txt
지현 : 회사 일이 잘 마무리되도록
김진수: 가족 건강을 위해
주원 - 시험 준비를 위해
민지) 가족과의 관계 회복을 위해
하은/ 새로운 직장 적응을 위해
```

---

## 4.2 줄 단위 입력

기본적으로 한 줄에 한 사람의 기도제목이 있다고 가정한다.

```txt
지현 : 회사 일
진수 : 가족 건강
주원 : 시험 준비
```

각 줄은 독립적으로 파싱한다.

---

## 4.3 여러 줄 기도제목

기도제목 내용이 여러 줄일 수 있다.

이 경우, 이름 패턴이 새로 등장하기 전까지는 이전 사람의 기도제목에 이어 붙인다.

예시 입력:

```txt
지현 : 회사 일이 잘 마무리되도록
이번 주 프로젝트가 잘 끝나도록
진수 : 가족 건강을 위해
```

파싱 결과:

| 이름 | 기도제목                             |
| -- | -------------------------------- |
| 지현 | 회사 일이 잘 마무리되도록 이번 주 프로젝트가 잘 끝나도록 |
| 진수 | 가족 건강을 위해                        |

---

## 4.4 구분자가 없는 입력

아래처럼 구분자가 없는 입력은 오인식 위험이 있다.

```txt
지현 회사 일이 잘 마무리되도록
```

MVP에서는 구분자 없는 입력을 기본 지원하지 않는다.

다만 추후 개선 시 다음 조건에서 제한적으로 지원할 수 있다.

* 줄의 첫 단어가 선택된 셀 인원의 이름 또는 별칭과 완전 일치하는 경우
* 동명이인이 없는 경우
* 사용자가 저장 전 확인하는 경우

---

# 5. 이름 정규화 규칙

## 5.1 공백 제거

이름 매칭 시 앞뒤 공백과 중간 공백을 제거한다.

```txt
장 지현 → 장지현
 지현  → 지현
```

---

## 5.2 호칭 제거

이름 끝에 붙은 호칭은 제거한다.

제거 대상:

```txt
님
자매
형제
집사
목사
목사님
전도사
전도사님
리더
리더님
선생님
```

예시:

```txt
지현자매 → 지현
진수형제 → 진수
민정님 → 민정
영호집사 → 영호
```

---

## 5.3 특수문자 제거

이름 주변의 특수문자는 제거한다.

대상 예시:

```txt
[]
()
{}
<>
*
-
_
.
,
```

예시:

```txt
[지현] → 지현
(진수) → 진수
*주원* → 주원
```

---

## 5.4 영문 대소문자

영문 이름 또는 별칭이 있을 수 있으므로 영문은 소문자로 비교한다.

```txt
Zia → zia
zia → zia
```

---

# 6. 성도 이름 데이터 규칙

## 6.1 members.name_aliases

이름 매칭에는 `members.name_aliases`를 사용한다.

예시:

| full_name | name_aliases |
| --------- | ------------ |
| 장지현       | 장지현,지현,지현자매  |
| 김진수       | 김진수,진수,진수형제  |
| 박주원       | 박주원,주원       |

---

## 6.2 별칭 자동 생성 권장

성도 등록 시 기본 별칭을 자동 생성하는 것을 권장한다.

예:

```txt
full_name = 장지현
자동 별칭 = 장지현, 지현
```

한국 이름 기준으로 전체 이름이 3글자 이상이면 마지막 2글자를 이름 별칭으로 추가할 수 있다.

예:

```txt
장지현 → 지현
김민수 → 민수
박하은 → 하은
```

단, 외국 이름, 2글자 이름, 특수한 이름은 수동 보정이 필요하다.

---

# 7. 매칭 우선순위

기도제목 입력 이름은 선택된 셀 인원 목록을 기준으로 다음 순서로 매칭한다.

## 7.1 1순위: 전체 이름 완전 일치

입력 이름이 `members.full_name`과 완전히 일치하면 매칭한다.

예:

```txt
입력: 장지현
성도: 장지현
결과: matched
```

---

## 7.2 2순위: 별칭 완전 일치

입력 이름이 `members.name_aliases` 중 하나와 완전히 일치하면 매칭한다.

예:

```txt
입력: 지현
별칭: 장지현,지현,지현자매
결과: matched
```

---

## 7.3 3순위: 전체 이름 끝부분 일치

입력 이름이 성도 전체 이름의 끝부분과 일치하면 매칭한다.

예:

```txt
입력: 지현
성도: 장지현
결과: matched
```

단, 이 규칙은 후보가 정확히 1명일 때만 자동 매칭한다.

---

## 7.4 후보가 여러 명인 경우

동일한 입력 이름에 대해 후보가 2명 이상이면 `ambiguous`로 처리한다.

예:

```txt
입력: 지현
후보: 장지현, 김지현
결과: ambiguous
```

자동 저장하지 않고 사용자에게 선택을 요청한다.

---

## 7.5 후보가 없는 경우

후보가 없으면 `unmatched`로 처리한다.

예:

```txt
입력: 민수
선택된 셀 인원에 민수 없음
결과: unmatched
```

사용자는 직접 성도를 선택하거나 해당 항목을 제외할 수 있다.

---

# 8. 파싱 결과 상태

파싱 결과는 다음 상태 중 하나를 가진다.

| status    | 설명             |
| --------- | -------------- |
| matched   | 성도 1명과 자동 매칭됨  |
| ambiguous | 후보가 2명 이상임     |
| unmatched | 후보를 찾지 못함      |
| invalid   | 입력 형식이 올바르지 않음 |

---

# 9. 결과 데이터 구조

## 9.1 API 또는 함수 응답 구조

```ts
type PrayerParseResult = {
  items: MatchedPrayerItem[];
  ambiguous: AmbiguousPrayerItem[];
  unmatched: UnmatchedPrayerItem[];
  invalid: InvalidPrayerItem[];
};

type MatchedPrayerItem = {
  input_name: string;
  member_id: string;
  matched_name: string;
  prayer_request: string;
  status: "matched";
  confidence: number;
};

type AmbiguousPrayerItem = {
  input_name: string;
  prayer_request: string;
  status: "ambiguous";
  candidates: {
    member_id: string;
    display_name: string;
    full_name: string;
  }[];
};

type UnmatchedPrayerItem = {
  input_name: string;
  prayer_request: string;
  status: "unmatched";
};

type InvalidPrayerItem = {
  raw_line: string;
  reason: string;
  status: "invalid";
};
```

---

## 9.2 matched 예시

```json
{
  "input_name": "지현",
  "member_id": "mem_001",
  "matched_name": "장지현",
  "prayer_request": "회사 일이 잘 마무리되도록",
  "status": "matched",
  "confidence": 1
}
```

---

## 9.3 ambiguous 예시

```json
{
  "input_name": "지현",
  "prayer_request": "회사 일이 잘 마무리되도록",
  "status": "ambiguous",
  "candidates": [
    {
      "member_id": "mem_001",
      "display_name": "장지현",
      "full_name": "장지현"
    },
    {
      "member_id": "mem_002",
      "display_name": "김지현",
      "full_name": "김지현"
    }
  ]
}
```

---

## 9.4 unmatched 예시

```json
{
  "input_name": "민수",
  "prayer_request": "가족 건강을 위해",
  "status": "unmatched"
}
```

---

## 9.5 invalid 예시

```json
{
  "raw_line": "이번 주 모두 건강하게 지내도록",
  "reason": "이름과 기도제목을 구분할 수 없습니다.",
  "status": "invalid"
}
```

# 10. 구분자 처리 규칙

## 10.1 지원 구분자

다음 구분자를 지원한다.

| 구분자 | 예시        |
| --- | --------- |
| `:` | 지현: 기도제목  |
| `：` | 지현：기도제목   |
| `-` | 지현 - 기도제목 |
| `)` | 지현) 기도제목  |
| `/` | 지현/ 기도제목  |

---

## 10.2 구분자 우선순위

한 줄에 여러 구분자가 있을 경우 가장 앞에 등장한 지원 구분자를 기준으로 분리한다.

예:

```txt
지현 : 회사 - 프로젝트가 잘 되도록
```

결과:

```txt
이름 = 지현
기도제목 = 회사 - 프로젝트가 잘 되도록
```

---

# 11. 중복 입력 처리

같은 사람이 여러 번 등장할 수 있다.

예:

```txt
지현 : 회사 일
지현 : 가족 건강
```

MVP에서는 같은 사람의 기도제목을 합친다.

결과:

```txt
지현 : 회사 일
가족 건강
```

UI에서는 다음 안내를 표시할 수 있다.

```txt
지현님의 기도제목이 2개 입력되어 하나로 합쳤습니다.
```

총 5건 중 3건 자동 매칭, 1건 확인 필요, 1건 미매칭

# 13. 저장 규칙

## 13.1 저장 대상

최종 저장 시 다음 컬럼에 반영한다.

`weekly_member_records`:

```txt
prayer_request
prayer_source_text
prayer_parsed_by
prayer_parse_confidence
```

---

## 13.2 저장 조건

저장 가능한 항목:

* `matched`
* 사용자가 후보를 선택한 `ambiguous`
* 사용자가 직접 성도를 선택한 `unmatched`

저장 불가 항목:

* 성도 선택이 완료되지 않은 `ambiguous`
* 성도 선택이 완료되지 않은 `unmatched`
* 수정되지 않은 `invalid`

---

## 13.3 prayer_parsed_by 값

| 상황          | 값      |
| ----------- | ------ |
| 사용자가 직접 입력  | manual |
| 규칙 기반 자동 분리 | rule   |
| AI 기반 분리    | ai     |

MVP에서는 `ai`를 사용하지 않는다.

---

## 13.4 prayer_parse_confidence 값

| 상황           | 값       |
| ------------ | ------- |
| 전체 이름 완전 일치  | 1       |
| 별칭 완전 일치     | 1       |
| 끝부분 일치 단일 후보 | 0.9     |
| 사용자가 직접 선택   | 0.8     |
| 직접 입력        | 비움 또는 1 |

---

# 14. 예외 케이스

## 14.1 빈 입력

입력값이 비어 있으면 파싱하지 않는다.

메시지:

```txt
기도제목을 입력해주세요.
```

---

## 14.2 이름만 있고 기도제목 없음

예:

```txt
지현 :
```

처리:

* invalid로 분류
* 메시지 표시

```txt
기도제목 내용이 비어 있습니다.
```

---

## 14.3 기도제목만 있고 이름 없음

예:

```txt
회사 일이 잘 되도록
```

처리:

* invalid로 분류
* 메시지 표시

```txt
이름을 찾을 수 없습니다.
```

---

## 14.4 동명이인

동명이인은 자동 저장하지 않는다.

반드시 사용자 선택이 필요하다.

---

## 14.5 선택된 셀에 없는 이름

선택된 셀에 없는 이름은 unmatched로 처리한다.

Admin도 특정 셀 리포트를 작성하는 경우, 선택된 셀 인원 기준으로만 매칭한다.

---

# 15. 구현 함수 제안

## 15.1 주요 함수

```ts
parsePrayerText(rawText: string, members: Member[]): PrayerParseResult

splitPrayerLines(rawText: string): ParsedLine[]

normalizeName(name: string): string

matchMemberByName(inputName: string, members: Member[]): MatchResult

mergeDuplicatePrayerItems(items: MatchedPrayerItem[]): MatchedPrayerItem[]
```

---

## 15.2 Member 타입

```ts
type Member = {
  member_id: string;
  full_name: string;
  display_name: string;
  name_aliases: string[];
};
```

---

## 15.3 MatchResult 타입

```ts
type MatchResult =
  | {
      status: "matched";
      member: Member;
      confidence: number;
    }
  | {
      status: "ambiguous";
      candidates: Member[];
    }
  | {
      status: "unmatched";
    };
```

# 17. MVP 제외 범위

MVP에서는 다음을 구현하지 않는다.

* 외부 AI 기반 자연어 분석
* 카카오톡 대화 전체 분석
* 음성 입력 분석
* 이미지 캡처 OCR
* 구분자 없는 복잡한 문장 분석
* 여러 셀 인원을 동시에 대상으로 한 매칭

---

# 18. 향후 개선 가능성

향후 다음 기능을 추가할 수 있다.

* AI 기반 기도제목 분리
* 카카오톡 대화 복사본 분석
* 구분자 없는 입력 지원
* 이름 오타 추천
* 자주 등장하는 별칭 자동 추가
* 기도제목 카테고리 자동 분류
* 민감정보 감지 및 경고

# 19. 나눔 일괄 입력 적용

본 문서의 이름 정규화, 선택된 셀 기준 매칭, 동명이인, 미매칭, 여러 줄 및 중복 입력 규칙은 개인 나눔 요약 일괄 입력에도 동일하게 적용한다.

구현 원칙:

* 공통 사람별 내용 parser를 사용한다.
* 기도제목과 나눔은 서로 다른 입력 및 결과 상태를 유지한다.
* 나눔 자동 분리 결과도 사용자 확인 전에는 개인별 입력란이나 저장 데이터에 반영하지 않는다.
* 나눔 내용은 민감 정보로 취급하며 외부 AI API에 전송하지 않는다.

# 20. Phase 2 구현 상태

`lib/prayer-parser.ts`와 공통 사람별 내용 parser는 다음 규칙을 테스트한다.

```txt
전체 이름, 별칭, 이름 끝부분 매칭 우선순위
선택된 셀 인원 목록 안에서만 매칭
호칭, 공백, 특수문자, 영문 대소문자 정규화
:, ：, -, ), / 구분자
여러 줄 내용과 중복 이름 병합
ambiguous, unmatched, invalid 분류
기도제목 전용 invalid 안내 문구
```

파싱 결과는 자동 저장하지 않으며 기존 UI의 사용자 확인 단계를 유지한다.

# 21. Phase 3-2 서버 API 적용

`parsePrayerRequests` Apps Script API는 다음 규칙을 적용한다.

```txt
1. 서버에서 요청 사용자의 활성 상태와 role을 확인한다.
2. Admin은 모든 활성 셀, cell_leader는 현재 활성 배정 셀만 선택할 수 있다.
3. 선택된 cell_id의 status = active 성도만 매칭 후보로 사용한다.
4. members.name_aliases 쉼표 문자열을 배열로 변환한 뒤 매칭한다.
5. matched / ambiguous / unmatched / invalid 결과만 반환한다.
6. parser 결과와 raw_text를 저장하지 않는다.
```

Google OAuth 신원 검증과 저장 전 사용자 확인이 포함된 실제 프론트엔드 연결은 후속 Phase에서 진행한다.
