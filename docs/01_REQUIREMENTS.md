# BLC Care 요구사항 정의서

## 1. 프로젝트 개요

### 1.1 프로젝트명

**BLC Care**

### 1.2 프로젝트 설명

BLC Care는 Bluelight 홍대교회의 셀 출결, 나눔 요약, 기도제목, 지원 제안, 새신자 정보, 장기결석자 현황을 관리하기 위한 웹 기반 교회 돌봄 관리 시스템이다.

셀리더는 담당 셀의 주차별 셀 모임 정보를 입력하고, Admin은 권한 범위 내에서 성도와 셀의 출결 및 돌봄 히스토리를 확인할 수 있다.

교회명, 셀 수, Admin 수, 셀리더 수, 성도 수는 운영 상황에 따라 변경될 수 있으므로, 시스템은 확장 가능한 구조를 전제로 설계한다.

### 1.3 프로젝트 목적

* 셀리더의 출결 및 기도제목 정리 부담을 줄인다.
* 성도별 출결, 나눔, 기도제목, 특이사항 히스토리를 누적 관리한다.
* 새신자 정보를 체계적으로 수집하고 성도 DB로 전환할 수 있게 한다.
* 장기결석자를 자동으로 확인하여 Admin이 돌봄 상황을 놓치지 않도록 한다.
* Google 기반 도구를 활용하여 관리자가 파일을 보관하고 백업하기 쉽게 한다.

## 2. 사용자 유형

### 2.1 Admin

Admin은 교회 운영 상황에 따라 1명 이상 지정될 수 있다.

Admin은 다음 기능을 사용할 수 있다.

* 전체 성도 조회
* 전체 셀 조회
* 전체 셀 리포트 조회
* 성도 상세 정보 조회 및 수정
* 성도 휴면 처리
* 새신자 DB 조회
* 새신자 정보를 성도 DB로 전환
* 장기결석자 조회
* 셀리더 계정 관리
* 셀 관리
* 시스템 설정 관리
* 데이터 백업 및 내보내기

### 2.2 셀리더

셀리더는 교회 셀 운영 상황에 따라 여러 명 지정될 수 있다.

셀리더는 다음 기능을 사용할 수 있다.

* 구글 로그인
* 본인에게 배정된 셀 인원 조회
* 본인 담당 셀의 주차별 리포트 작성
* 해당 주차 동안 리포트 수정
* 본인 담당 셀 성도의 출결, 나눔 요약, 기도제목 히스토리 조회
* 본인 담당 셀 성도의 특이사항 작성 및 해결 여부 체크

한 명의 셀리더는 하나 이상의 셀을 담당할 수 있다.
셀 담당 정보는 사용자 정보에 직접 저장하지 않고, 별도의 셀 배정 정보로 관리한다.

### 2.3 새신자

새신자는 로그인하지 않고 새신자 등록폼을 제출할 수 있다.

새신자는 다음 정보를 입력할 수 있다.

* 이름
* 휴대폰 번호
* 주소
* 방문동기
* 방문경로
* 신앙생활 경험
* 예배 이후 일정
* 개인정보 수집 동의

## 3. 핵심 기능

## 3.1 구글 로그인

### 요구사항

* Admin과 셀리더는 구글 계정으로 로그인한다.
* 로그인한 이메일을 기준으로 사용자 권한을 확인한다.
* 사용자의 role에 따라 접근 가능한 화면을 제한한다.

### 사용자 역할

사용자는 하나 이상의 역할을 가질 수 있다.

예를 들어 한 사용자가 `admin` 역할과 `cell_leader` 역할을 동시에 가질 수 있다.

| role | 설명 |
|---|---|
| admin | 전체 관리 권한 |
| cell_leader | 담당 셀 관리 권한 |

역할은 `roles` 값으로 관리한다.

예시:

- admin
- cell_leader
- admin,cell_leader

### 권한 규칙

* Admin은 모든 셀과 모든 성도를 조회할 수 있다.
* 셀리더는 본인이 담당하는 셀의 정보만 조회할 수 있다.
* 새신자 등록폼은 로그인 없이 접근 가능하다.
* 향후 필요 시 부서 관리자, 교역자, 보조 리더 등 추가 role을 확장할 수 있어야 한다.

### 복수 역할 규칙

- 한 사용자는 여러 role을 가질 수 있다.
- `admin`과 `cell_leader` 역할을 동시에 가진 사용자는 전체 관리 기능과 담당 셀 관리 기능을 모두 사용할 수 있다.
- `admin` 권한이 있는 사용자는 전체 데이터에 접근할 수 있다.
- `cell_leader` 권한이 있는 사용자는 본인에게 배정된 셀의 데이터에 접근할 수 있다.
- 복수 역할 사용자는 기본적으로 Admin 대시보드로 이동하되, 상단 메뉴에서 셀리더 기능으로 이동할 수 있다.

성도 정보를 등록, 조회, 수정할 수 있어야 한다.

### 성도 기본 정보

* 사진
* 이름
* 이름 별칭
* 나이
* 생년월일
* 휴대폰 번호
* 첫방문일
* 등록일
* 거주지
* 직장
* 직업
* 직책
* 믿음 시작 년도
* 학습 여부
* 세례 여부
* 가족관계
* 현재 소속 셀
* 상태

  * 활동
  * 휴면
  * 이탈 또는 기타
* 메모

### 목록 화면 요구사항

* 전체 성도 목록을 볼 수 있어야 한다.
* 셀 기준으로 필터링할 수 있어야 한다.
* 이름으로 검색할 수 있어야 한다.
* 이름순으로 정렬할 수 있어야 한다.
* 상태별로 필터링할 수 있어야 한다.
* 성도 항목을 클릭하면 개인 상세 페이지로 이동한다.
* 인원이 늘어날 수 있으므로 검색, 필터, 페이지네이션을 고려한다.

### 개인 상세 페이지 요구사항

개인 상세 페이지에서는 다음 정보를 확인할 수 있어야 한다.

* 기본 인적사항
* 현재 셀
* 상태
* 출결 히스토리
* 나눔 요약 히스토리
* 기도제목 히스토리
* 지원 제안 히스토리
* 특이사항 목록
* 특이사항 해결 여부

## 3.3 셀 관리

### 요구사항

* 셀은 Admin이 관리한다.
* 각 셀에는 셀리더가 지정된다.
* 성도는 특정 셀에 소속된다.
* 셀은 추가, 수정, 비활성화될 수 있다.
* 성도의 소속 셀은 변경될 수 있다.
* 셀 변경 이력은 보존되어야 한다.

### 셀 변경 이력

성도의 셀이 변경될 경우 다음 정보가 기록되어야 한다.

* 성도 ID
* 이전 셀
* 변경 셀
* 변경 시작일
* 변경 종료일
* 변경 사유
* 변경 처리자

## 3.4 주차별 셀 리포트

### 요구사항

셀리더는 주차별로 셀 리포트를 작성할 수 있어야 한다.

셀 리포트에는 다음 정보가 포함된다.

* 주차
* 셀
* 작성자
* 전체 나눔 요약
* 인원별 출결
* 인원별 나눔 요약
* 인원별 기도제목
* 인원별 지원 제안

### 수정 규칙

* 셀리더는 해당 주차 동안 리포트를 수정할 수 있다.
* 해당 주차가 지난 뒤에는 기본적으로 수정할 수 없다.
* 주차 종료 후 수정은 Admin만 가능하다.
* 추후 설정에 따라 수정 가능 기간을 변경할 수 있어야 한다.

### 저장 규칙

셀리더가 하나의 셀 리포트를 저장하면, 각 인원별 기록이 자동으로 개인 히스토리에 누적되어야 한다.

예시:

* 특정 주차의 셀 리포트 작성
* 성도별 출석 여부 입력
* 성도별 나눔 요약 입력
* 성도별 기도제목 입력
* 성도별 지원 제안 입력

저장 결과:

* 셀 리포트 1건 생성
* 개인별 기록 생성
* 각 개인 상세 페이지에서 히스토리 확인 가능

## 3.5 기도제목 일괄 입력 및 자동 분리

### 기능 목적

셀리더가 여러 사람의 기도제목을 한 번에 복사해 붙여넣으면, 시스템이 사람별로 자동 분리하여 각 성도의 기도제목 입력칸에 넣어준다.

### 입력 예시

```txt
지현 : 회사 일이 잘 마무리되도록 기도해주세요.
김진수 : 가족 건강을 위해 기도해주세요.
주원자매 : 시험 준비를 위해 기도해주세요.
```

### 지원해야 하는 입력 형식

```txt
이름 : 기도제목
이름 - 기도제목
이름) 기도제목
이름/ 기도제목
```

### 이름 매칭 규칙

성도 DB에는 이름 별칭을 저장한다.

예시:

| full_name | name_aliases  |
| --------- | ------------- |
| 장지현       | 장지현, 지현, 지현자매 |
| 김진수       | 김진수, 진수, 진수형제 |
| 박주원       | 박주원, 주원       |

기도제목 자동 분리 시 다음 순서로 매칭한다.

1. 현재 셀 인원 중 전체 이름 완전 일치
2. 현재 셀 인원 중 이름 별칭 완전 일치
3. 현재 셀 인원 중 전체 이름의 끝 글자 일치
4. 동명이인 또는 중복 후보가 있으면 저장하지 않고 사용자에게 선택 요청
5. 매칭되지 않은 이름은 미매칭 목록으로 표시

### 호칭 제거 규칙

다음 호칭은 이름 매칭 시 제거한다.

* 님
* 자매
* 형제
* 집사
* 목사
* 전도사
* 리더

예시:

```txt
지현자매 → 지현
진수형제 → 진수
민정님 → 민정
```

### 저장 전 확인

자동 분리 결과는 바로 저장하지 않는다.

반드시 셀리더가 결과를 확인한 뒤 저장한다.

### 분리 결과 화면

분리 결과는 다음 형태로 보여준다.

| 입력 이름 | 매칭 결과 | 기도제목           | 상태    |
| ----- | ----- | -------------- | ----- |
| 지현    | 장지현   | 회사 일이 잘 마무리되도록 | 자동 매칭 |
| 진수    | 김진수   | 가족 건강          | 자동 매칭 |
| 주원    | 박주원   | 시험 준비          | 자동 매칭 |

동명이인 발생 시:

```txt
'지현'에 해당하는 인원이 2명 있습니다.
장지현 / 김지현 중 선택해주세요.
```

## 3.6 특이사항 관리

### 요구사항

Admin과 셀리더는 성도별 특이사항을 기록할 수 있다.

특이사항에는 다음 정보가 포함되어야 한다.

* 성도 ID
* 특이사항 내용
* 기록일
* 기록자
* 해결 여부
* 해결일
* 해결 처리자

### 해결 체크

* 특이사항마다 체크박스를 둔다.
* 체크하면 해결 처리된다.
* 해결 처리 시 해결일과 해결 처리자가 기록된다.
* 미해결 특이사항만 필터링할 수 있어야 한다.

## 3.7 새신자 등록

### 요구사항

새신자는 로그인 없이 등록폼을 제출할 수 있다.

### 새신자 입력 항목

* 이름
* 휴대폰 번호
* 주소
* 방문동기
* 방문경로
* 신앙생활 경험
* 예배 이후 일정
* 개인정보 수집 동의

### Admin 기능

Admin은 새신자 DB에서 다음 작업을 할 수 있다.

* 새신자 목록 조회
* 새신자 상세 조회
* 연락 상태 변경
* 새신자 정보를 성도 DB로 전환
* 전환 완료 시 새신자 상태를 converted로 변경

### 새신자 상태

| status    | 설명          |
| --------- | ----------- |
| new       | 신규 제출       |
| contacted | 연락 완료       |
| converted | 성도 DB 전환 완료 |
| archived  | 보관          |

## 3.8 장기결석자 관리

### 요구사항

N개월 이상 출석 기록이 없는 성도를 장기결석자로 표시한다.

출석 완료 기록이 한 번도 없는 성도는 등록일을 기준으로 계산하고, 등록일도 없으면 첫방문일을 기준으로 계산한다. 세 날짜가 모두 없으면 자동 장기결석 계산 대상에서 제외하고 Admin 확인 대상으로 남긴다.

기본 기준은 3개월로 하되, Admin 설정에서 변경 가능해야 한다.

### 기능

* 장기결석자 자동 계산
* Admin 대시보드에 알림 표시
* 장기결석자 전용 페이지 제공
* 마지막 출석일 표시
* 결석 기간 표시
* 조치 메모 작성
* 확인 처리 가능

### 장기결석자 목록 항목

* 이름
* 현재 셀
* 마지막 출석일
* 결석 기간
* 상태
* 조치 메모
* 확인 여부

## 3.9 사진 등록

### 요구사항

* 성도별 사진을 등록할 수 있어야 한다.
* 사진 파일은 Google Drive에 저장한다.
* 성도 DB에는 사진 URL 또는 파일 ID를 저장한다.
* 사진은 성도 목록과 개인 상세 페이지에서 표시한다.

## 3.10 백업 및 로그

### 백업 요구사항

* Google Sheets 파일은 관리자가 직접 보관할 수 있어야 한다.
* 정기적으로 CSV 또는 XLSX 형태로 백업할 수 있어야 한다.
* Google Drive에 백업 폴더를 둘 수 있어야 한다.

### 로그 요구사항

중요 데이터 변경 시 audit log를 남긴다.

기록 대상:

* 성도 정보 생성/수정/삭제
* 휴면 처리
* 셀 변경
* 셀 리포트 생성/수정
* 특이사항 생성/해결
* 새신자 성도 전환
* 장기결석자 확인 처리

로그 항목:

* log_id
* action
* target_type
* target_id
* changed_by
* changed_at
* before_value
* after_value

## 7. 비기능 요구사항

## 7.1 반응형 지원

* PC와 모바일에서 모두 사용할 수 있어야 한다.
* 셀리더는 모바일에서 리포트를 작성할 수 있어야 한다.
* Admin은 PC에서 전체 관리가 편해야 한다.

## 7.2 개인정보 보호

본 시스템은 성도 개인정보, 기도제목, 신앙생활 정보 등 민감한 정보를 다룬다.

따라서 다음 원칙을 따른다.

* 로그인 사용자는 허가된 사용자만 등록한다.
* 셀리더는 본인 담당 셀 정보만 접근한다.
* Admin은 전체 정보에 접근할 수 있다.
* 새신자 등록폼에는 개인정보 수집 동의 체크박스를 둔다.
* 외부 AI API에는 개인정보를 전송하지 않는 것을 기본 원칙으로 한다.
* 기도제목 자동 분리는 1차 버전에서 규칙 기반으로 처리한다.
* 실제 성도 데이터는 개발 중 사용하지 않고, mock data로 개발한다.

## 7.3 성능 및 확장 기준

본 시스템은 성도, Admin, 셀리더, 셀 수가 늘어날 수 있음을 전제로 한다.

Google Sheets 기반 구현의 응답 지연 문제로 Supabase PostgreSQL 전환을 결정했다.

* 성도 목록은 검색, 필터, 페이지네이션을 지원한다.
* 셀 리포트와 개인 히스토리는 필요한 범위만 조회한다.
* 장기결석자 계산은 전체 데이터를 매번 과도하게 조회하지 않도록 설계한다.
* 관계 조회와 주요 필터 컬럼에 PostgreSQL 인덱스를 적용한다.
* 모든 민감 테이블에 RLS를 적용한다.

## 7.4 백업 가능성

관리자가 PostgreSQL 백업을 생성하고 복구 절차를 확인할 수 있어야 한다.

## 11. 성공 기준

BLC Care의 1차 성공 기준은 다음과 같다.

* 셀리더가 모바일에서 이번 주 셀 리포트를 작성할 수 있다.
* 셀리더가 나눔과 기도제목을 한 번에 붙여넣고 사람별로 자동 분리할 수 있다.
* 성도별 출결, 나눔, 기도제목 히스토리가 자동으로 쌓인다.
* Admin이 전체 성도와 셀별 현황을 확인할 수 있다.
* Admin이 새신자 정보를 확인하고 성도 DB로 전환할 수 있다.
* Admin이 장기결석자를 확인할 수 있다.
* 모든 핵심 데이터는 Google Sheets와 Google Drive에 저장되어 백업 가능하다.

## 12. 나눔 일괄 입력 확장

기도제목 자동 분리에 사용하는 선택 셀 기준 이름 매칭 규칙을 개인 나눔 요약에도 동일하게 적용한다.

* 나눔과 기도제목은 각각 별도의 일괄 입력 영역을 사용한다.
* 자동 분리 결과는 바로 저장하지 않고 사용자가 확인한 뒤 개인별 입력란에 반영한다.
* 나눔 내용도 민감 정보로 취급하며 외부 AI API에 전송하지 않는다.
