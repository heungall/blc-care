# BLC Care Design System

## 1. 문서 개요

본 문서는 BLC Care의 프론트엔드 디자인 방향, UI 원칙, 색상, 레이아웃, 컴포넌트 스타일 기준을 정의한다.

BLC Care는 교회 출결 및 성도 돌봄 관리 시스템이다.
따라서 디자인은 단순한 관리자 페이지가 아니라, **차분하고 따뜻하며 신뢰감 있는 돌봄 도구**를 지향한다.

---

# 2. 디자인 방향

## 2.1 핵심 키워드

```txt
calm
warm
clear
pastoral care
mobile-friendly
soft admin
```

한국어 기준:

```txt
차분한
따뜻한
정돈된
돌봄 중심
모바일 친화적
부드러운 관리자 도구
```

## 2.2 디자인 목표

BLC Care의 UI는 다음 목표를 가진다.

```txt
1. 셀리더가 모바일에서 쉽게 입력할 수 있어야 한다.
2. Admin이 PC에서 전체 현황을 빠르게 파악할 수 있어야 한다.
3. 개인정보와 기도제목을 다루는 앱답게 과하게 화려하지 않아야 한다.
4. 사용자가 부담을 느끼지 않도록 부드럽고 친절한 화면을 제공해야 한다.
5. 기능이 많아져도 복잡해 보이지 않도록 카드, 배지, 섹션을 적극 사용한다.
```

# 3. Color Palette

## 3.1 기본 색상 방향

Bluelight 홍대교회의 이름에서 오는 Blue 계열을 중심으로 사용한다.

단, 너무 강한 파랑이나 차가운 관리자 도구 느낌은 피한다.
부드러운 블루, 쿨톤 회백색, 화이트 카드, 세이지/앰버/로즈 계열 상태 색상을 사용한다.

## 3.3 사용 원칙

* Primary Blue는 버튼과 주요 액션에만 사용한다.
* 배경은 `#F8FAFC` 계열로 부드럽게 유지한다.
* 카드 배경은 흰색을 기본으로 한다.
* 위험/주의 색상은 강한 빨강보다 부드러운 rose/amber 계열을 사용한다.
* 기도제목, 특이사항 등 민감 정보는 과한 강조색을 쓰지 않는다.

# 4. Typography

## 4.1 기본 원칙

* 가독성을 최우선으로 한다.
* 모바일 입력 화면에서는 글자를 너무 작게 만들지 않는다.
* Admin 화면에서는 정보 밀도를 높이되, 섹션 구분을 분명히 한다.

## 4.2 추천 크기

| 용도            | 크기                       |
| ------------- | ------------------------ |
| Page Title    | `text-2xl` 또는 `text-3xl` |
| Section Title | `text-lg` 또는 `text-xl`   |
| Card Title    | `text-base` 또는 `text-lg` |
| Body          | `text-sm` 또는 `text-base` |
| Helper Text   | `text-sm`                |
| Badge         | `text-xs`                |

## 4.3 문구 톤

문구는 짧고 친절하게 작성한다.

좋은 예:

```txt
이번 주 리포트를 작성해주세요.
기도제목을 한 번에 붙여넣을 수 있어요.
저장 전 내용을 확인해주세요.
```

피할 예:

```txt
데이터가 존재하지 않습니다.
유효하지 않은 요청입니다.
처리 불가능한 상태입니다.
```

# 5. Layout Principles

## 5.1 전체 레이아웃

로그인 후 화면은 다음 구조를 기본으로 한다.

PC:

```txt
Sidebar + Header + Main Content
```

Mobile:

```txt
Top Header + Bottom Navigation + Card Content
```

## 5.2 Admin 화면

Admin 화면은 PC 사용성을 우선한다.

원칙:

* 좌측 사이드바 사용
* 대시보드는 카드형 요약 사용
* 목록은 테이블 기반
* 상세 정보는 섹션 단위로 구분
* 필터는 상단에 배치

## 5.3 셀리더 화면

셀리더 화면은 모바일 사용성을 우선한다.

원칙:

* 하단 탭 또는 간단한 상단 메뉴 사용
* 리포트 작성은 카드형 입력
* 큰 버튼 사용
* 입력 필드는 충분한 높이 확보
* 한 화면에 너무 많은 정보를 몰아넣지 않기

민감 정보가 많으므로, 섹션을 명확히 구분한다.

## 6.5 Weekly Report `/reports/new`

목적:

* 셀리더가 모바일에서 빠르게 출결과 기도제목을 기록한다.

구성:

```txt
주차 정보
셀 선택
1. 출결 입력
  출결 현황 요약
  전체 출석 일괄 선택
  인원별 출결 상태 버튼
2. 나눔 작성
  출결 입력 결과 요약
  전체 나눔 요약
  나눔·기도제목 일괄 입력
  자동 분리 결과
  인원별 기록 카드
[임시저장]
[제출]
```

출결 입력 단계는 인원별 긴 카드 대신 이름과 상태 버튼을 한 줄 단위로 촘촘하게 배치한다.
모바일과 데스크톱 모두 한 사람의 출석 선택과 상태 변경을 한 줄에 표시한다.
기본 입력은 출석한 사람만 체크하고, `미선택 인원 결석 처리`를 명시적으로 눌러 남은 미확인 인원을 결석으로 변경한다.
선택하지 않은 인원을 자동으로 결석 처리하지 않으며, 사유 결석과 미확인은 인원별 상태 선택에서 직접 변경한다.
나눔 작성 단계의 인원별 카드에는 선택된 출결 상태를 badge로 요약하고, `출결 수정`으로 이전 단계에 돌아갈 수 있게 한다.

인원별 기록 카드 예시:

```txt
장지현

[출석]

기도제목
[textarea]

나눔 요약
[textarea]

지원 제안
[textarea]
```

# 7. Components

## 7.1 Button

버튼 종류:

| Variant     | 용도       |
| ----------- | -------- |
| primary     | 주요 액션    |
| secondary   | 보조 액션    |
| outline     | 선택/이동    |
| ghost       | 가벼운 액션   |
| destructive | 삭제/위험 액션 |

원칙:

* 모바일 주요 버튼은 full width 가능
* 제출/저장은 primary
* 취소/뒤로가기는 secondary 또는 ghost
* 삭제/비활성화는 destructive

---

## 7.2 Card

카드는 다음 정보에 사용한다.

* 성도 요약
* 셀 요약
* 리포트 요약
* Admin 대시보드 지표
* 인원별 리포트 입력
* 특이사항

카드 스타일:

```txt
rounded-2xl
border
bg-white
shadow-sm
p-4 또는 p-6
```

| 휴면       | neutral      |
| 이탈/보관    | neutral      |
| 출석       | success      |
| 결석       | danger       |
| 사유결석     | warning      |

| 잠금       | neutral      |
| 장기결석     | warning      |

## 7.4 Form

폼 원칙:

* label은 명확하게 표시한다.
* 필수값은 `*` 표시한다.
* helper text를 적극 사용한다.
* 오류 메시지는 필드 아래에 표시한다.
* textarea는 모바일에서 충분히 크게 만든다.

좋은 예:

```txt
기도제목
여러 줄로 입력해도 괜찮아요.
```

# 8. Prayer Parser UI

기도제목 자동 분리 UI는 BLC Care의 핵심 경험이다.

## 8.1 입력 영역

```txt
기도제목 일괄 입력

예시)
지현 : 회사 일이 잘 마무리되도록
김진수 : 가족 건강을 위해
주원자매 : 시험 준비를 위해

[자동 분리하기]
```

## 8.2 결과 요약

```txt
총 5건 중 3건 자동 매칭, 1건 확인 필요, 1건 미매칭
```

## 8.3 결과 카드

matched:

```txt
지현 → 장지현
회사 일이 잘 마무리되도록
```

ambiguous:

```txt
'지현'에 해당하는 인원이 2명 있어요.
저장할 사람을 선택해주세요.

[장지현] [김지현]
```

unmatched:

```txt
'민수'를 선택된 셀 인원에서 찾을 수 없어요.
직접 선택하거나 제외해주세요.
```

## 8.4 저장 원칙

자동 분리 결과는 바로 저장하지 않는다.

사용자가 확인 후 리포트 저장 시 반영한다.

# 9. Responsive Rules

## 9.1 Breakpoint Strategy

권장 기준:

```txt
Mobile: < 768px
Tablet: 768px ~ 1024px
Desktop: >= 1024px
```

## 9.2 Mobile

* 하단 탭 사용 가능
* 주요 버튼은 full width 선호
* 성도 목록은 카드형
* 리포트 작성은 카드형
* 입력 필드 높이 넉넉히
* 테이블 지양

## 9.3 Desktop

* 사이드바 사용
* 목록은 테이블 가능
* 대시보드는 grid 카드
* 필터와 검색 영역을 상단에 배치

---

# 10. Accessibility

접근성 원칙:

```txt
1. 버튼과 입력 필드는 명확한 label을 가진다.
2. 색상만으로 상태를 구분하지 않는다.
3. 상태 배지는 텍스트를 함께 표시한다.
4. 클릭 영역은 모바일에서 충분히 크게 만든다.
5. placeholder에만 중요한 정보를 넣지 않는다.
6. 에러 메시지는 필드 가까이에 표시한다.
```

# 14. Design Implementation Rules

## 14.1 Tailwind

* 색상은 가능하면 디자인 토큰을 사용한다.
* 임의 hex 값을 컴포넌트 곳곳에 직접 쓰지 않는다.
* 반복되는 스타일은 컴포넌트화한다.

## 14.2 Layout

* 페이지마다 중복 레이아웃을 만들지 않는다.
* 공통 레이아웃 컴포넌트를 사용한다.
* 모바일과 데스크톱을 함께 고려한다.

## 14.3 State

상태 표현은 badge와 helper text를 함께 사용한다.

예:

```txt
[장기결석] 마지막 출석일: 2026-02-01
```

# 15. Codex UI Implementation Rules

Codex는 UI 구현 시 다음 원칙을 따른다.

```txt
1. docs/09_DESIGN_SYSTEM.md를 먼저 확인한다.
2. 새 화면은 기존 레이아웃과 컴포넌트 패턴을 재사용한다.
3. 셀리더 기능은 모바일 사용성을 우선한다.
4. Admin 기능은 PC 사용성을 우선하되 모바일에서도 깨지지 않게 한다.
5. 동일한 상태값은 동일한 Badge 스타일로 표시한다.
6. 민감 정보는 과하게 강조하지 않는다.
7. Prayer Parser UI는 저장 전 확인 흐름을 반드시 포함한다.
8. 접근성 원칙을 지킨다.
```

성도 상세의 직장, 직업, 직책은 기본 정보 카드 안에서 다른 개인 정보와 같은 시각적 비중으로 표시하며 목록 카드에는 표시하지 않는다.

# 16. MVP Design Acceptance Criteria

MVP 디자인 완료 기준은 다음과 같다.

```txt
1. 모바일에서 셀리더가 리포트를 작성할 수 있다.
2. 나눔 및 기도제목 일괄 입력 영역이 모바일에서 사용하기 쉽다.
3. 성도 목록이 모바일에서는 카드형, PC에서는 테이블 또는 카드형으로 보기 좋다.
4. Admin Dashboard에서 전체 현황이 카드로 명확히 보인다.
5. 상태값이 Badge로 일관되게 표현된다.
6. 권한 없음, 데이터 없음, 로딩, 에러 상태가 모두 정의되어 있다.
7. 색상과 버튼 스타일이 화면마다 흔들리지 않는다.
```

# 17. 사람별 일괄 입력 UI

나눔과 기도제목 일괄 입력은 동일한 카드 패턴을 사용한다.

* 각 카드의 제목과 안내 문구로 현재 입력 종류를 명확히 구분한다.
* 자동 매칭, 후보 선택 필요, 미매칭, 형식 확인 상태를 동일한 Badge 스타일로 표현한다.
* 각 입력 종류마다 별도의 확인 버튼을 제공한다.
* 모바일에서는 입력 영역과 확인 버튼을 전체 너비로 제공한다.
